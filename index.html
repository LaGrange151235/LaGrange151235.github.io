NIST Definition

Cloud Computing (NIST 2011)
“… a model for enabling ubiquitous, convenient, on-demand network access to a shared pool of configurable computing resources (e.g., networks, servers, storage, applications and services) that can be rapidly provisioned and released with minimal management effort or service provider interaction …

  • CHARACTERISTICS
    • On-demand self-service: no need to contact the service provider
    • Broad network access: can be accessed by the internet
    • Resource pooling: support the elastic
    • Rapid elasticity: realtime response
    • Measured service: pay per use
  • DEPLOYMENT MODELS
    • Private Cloud
    • Community Cloud
    • Public Cloud
    • Hybrid Cloud
  • SERVICE MODELS
    • Software-as-a-Service (SaaS) => software as the smallest element
    • Platform-as-a-Service (PaaS) => platform(maybe a set of softwares) as the smallest element
    • Infrastructure-as-a-Service (IaaS) => infrastructure(maybe a VM) as the smallest element

Cloud Characteristics

  1. On-demand self-service through a service portal
  2. Broad network access (ubiquitous(everywhere) access)
  3. Location-independent resource pooling (multi-tenancy)
  4. Rapid elasticity – time to market / fast deployment
  5. Measured service (pay-per-use)

Cloud Service(Delivery) Models

Infrastructure-as-a-Service (IaaS)

  • Virtual Server is an IaaS environment(always with a VM), it provides virtualized server resources.

Platform-as-a-Service (PaaS)

Software-as-a-Service (SaaS)

Service Models Pros Cons
on-premise highest flexibility long time-to-market
IaaS scalability, no hw procure privacy
PaaS DB, Framworks, middleware ready vendor lock in
SaaS time-to-market lowest flexibility

Conceptual Reference Architecture


5 Major Actors

  • Cloud consumer
    • maintains a business relationship with, and uses service from Cloud providers
  • Cloud provider
    • offers a cloud service to Cloud consumers
  • Cloud auditor
    • conducts independent assessment of cloud services, system operations, performance and security of the cloud implementation
  • Cloud broker
    • manages the use, performance and delivery of cloud services, and negotiates relationships between Cloud providers and Cloud consumers
  • Cloud carrier
    • provides connectivity and transport of cloud services from Cloud providers to Cloud consumers

Cloud Deployment Models

Private Cloud

  • solely for used by an organization
  • for enterprises/corporations with large scale IT

Public Cloud

  • available to general public, i.e., shared by all consumers
  • open market for on demand computing and IT resources
  • concerns: limited SLA, reliability, availability, security, trust

Community Cloud

  • shared by several organizations and supporting a specific community

Hybrid Cloud

  • two or more public and private clouds that interoperate
  • extends private cloud(s) to include a shared public cloud

MIT6.S081 Lab5 Copy-on-Write Fork for xv6

发表于 2022-07-10 分类于 自学

问题提出

在xv6中，系统调用fork()会将父进程的用户态内存复制到子进程。如果父进程很大，复制可能会消耗很长的一段时间。更糟糕的是，这些复制工作大部分是无效的，例如在fork()之后立刻在子进程中exec()导致子进程中抹去之前复制的内存，可能完全没有使用过之前复制的内存。另一方面，当父进程与子进程都使用同一页，并且至少其中一个需要对其写入，才需要真正的内存复制。

解决方案

对fork()实现copy-on-write(COW)功能来延迟内存申请和复制，直到子进程真正需要复制的新内存。
COW化的fork()只会为子进程建立一个新的页表，并将页表映射到父进程的物理内存上，此后令父进程和子进程的页入口(PTE)都设置为不可写的(clear PTE_W)。当任意一个进程对COW页进行写操作时，CPU会遇到页错误(page fault)。内核中的页错误处理函数检测到COW页错误，为发生页错误的进程申请内存并复制内容，将页入口设置为可写。从内核返回后，用户程序对页的写操作将不会在遇到页错误。
COW化的fork()使得物理内存释放更加复杂，应当使用数据结构记录多个进程对于一个物理地址的引用，直到所有的引用都结束再进行内存释放。

COW实现

相关变量与函数声明

修改uvmcopy()的复制与映射策略

在kenerl/vm.c中函数uvmcopy()函数中注释掉内存的申请与复制，将新页new[i]直接映射到父进程内存物理地址pa上，并将PTE_W位消去，将PTE_COW位设置，最后对pa对应的引用记录计数加一。

为usetrap()添加处理COW page fault的功能

查询文档，找到写页错误的scause寄存器值为15，对15增加处理判断语句，然后进行COW页错误检查，如果检查通过则对内存进行申请与复制

  1. COW页错误检查条件
    • 虚拟地址是否越界
      • 限制1：MAXVA最大虚拟地址限制
      • 限制2：p->sz进程内存大小
    • 页入口是否合法
      • 页入口是否存在
      • 页标记PTE_COW是否置位，是否为COW页
      • 页标记PTE_V是否置位，是否可用
      • 页标记PTE_U是否置位，是否为用户页
  2. 内存申请与复制流程
    1. 申请新的物理内存
    2. 设置PTE_W位，使页可写
    3. 消除PTE_COW位，取消页COW标记
    4. 重新映射页表

维护引用记录

首先声明一个数组用来维护对应地址的引用计数。get_reference_index()函数用来计算物理地址对应的引用记录下标，get_reference()函数用来获取物理地址对应的引用记录值。add_reference()函数用来增加引用计数值，sub_reference()函数用来减少引用计数值。

此后在freerange()中用零初始化引用记录。

修改kfree()函数，每次申请释放内存都减小引用计数，使当且仅当引用计数为0时才实际释放内存。

在kalloc()函数中用1初始化父进程内存的引用计数。

copyout()的COW化

在kernel/vm.c中copyout()函数中设置与uvmcopy()函数中类似的COW页检查和内存申请与复制，为copyout()实现COW化。

find

  • Hardness: moderate
  • Details: 实现文件查找，这里的查找需要支持文件夹之下内容的查找
  • Implementation: 这里主要是通过递归的调用find在给出的path中搜索target文件，搜索比对的方法是在path的字符串中切割出长度为target文件名长度的一部分进行比较，需要注意的是要跳过文件符’.’和’..’，他们分别指代当前文件夹路径和上级文件夹路径

xargs

  • Hardness: moderate
  • Details: xargs是Unix和类Unix系统中的一个重要指令，使用xargs的指令一般形式如下：
    $${commands}|xargs\quad{command}$$
    这里的|符号是管道pipe操作符，意思是将|左侧的指令执行的结果作为参数通过管道传递给管道右侧的指令，实现一个指令对多个参数的执行
  • Implementation: 这里主要是通过封装好的run()函数，通过fork()建立新的进程，分别对每一个读取到的参数执行命令
